## Overview

The Fisioterapia Backend uses a sophisticated user management system based on Mongoose discriminators. This allows different user types (patients, physiotherapists, and administrators) to share a common base schema while having type-specific fields.

## User Architecture

### Base User Schema

All users share a common schema defined in `src/models/usuarioModel.js`:

```javascript
const userSchema = mongoose.Schema({
  nombre: String,
  apellidoPaterno: String,
  apellidoMaterno: String,
  dni: {
    type: String,
    unique: true,
    required: true
  },
  cumpleanios: String,
  genero: String,
  img: String,
  celular: String,
  direccion: String,
  correo: {
    type: String,
    unique: true
  },
  password: {
    type: String,
    required: true
  },
  estado: {
    type: String,
    enum: ["activo", "inactivo"],
    default: "activo"
  }
}, { discriminatorKey: "rol", collection: "user" })
```

<Note>
All user types are stored in the same `user` collection, differentiated by the `rol` field (discriminator key).
</Note>

### User Types

<Tabs>
  <Tab title="Patient (Paciente)">
    Extends the base user schema with medical information:
    
    ```javascript
    const pacienteSchema = new mongoose.Schema({
      historialMedico: [String],
      alergias: [String]
    });
    
    export const pacienteModel = userModel.discriminator("paciente", pacienteSchema);
    ```
    
    **Additional Fields:**
    - `historialMedico` - Array of medical history entries
    - `alergias` - Array of known allergies
  </Tab>
  
  <Tab title="Physiotherapist (Fisioterapeuta)">
    Extends the base user schema with professional information:
    
    ```javascript
    const fisioterapeutaSchema = new mongoose.Schema({
      especialidad: String,
      colegiatura: String,
      universidad: String,
      descProfesional: String
    });
    
    export const fisioModel = userModel.discriminator("fisioterapeuta", fisioterapeutaSchema);
    ```
    
    **Additional Fields:**
    - `especialidad` - Specialization area
    - `colegiatura` - Professional license number
    - `universidad` - University/education institution
    - `descProfesional` - Professional description
  </Tab>
  
  <Tab title="Administrator (Admin)">
    Extends the base user schema with administrative fields:
    
    ```javascript
    const adminSchema = new mongoose.Schema({
      cargo: String
    });
    
    export const adminModel = userModel.discriminator("admin", adminSchema);
    ```
    
    **Additional Fields:**
    - `cargo` - Administrative position/role
  </Tab>
</Tabs>

## Creating Users

### Create a User

#### Endpoint

```http
POST /api/user
```

#### Request Body

```json
{
  "nombre": "Juan",
  "apellidoPaterno": "Perez",
  "apellidoMaterno": "Garcia",
  "dni": "12345678",
  "cumpleanios": "1990-05-15",
  "genero": "masculino",
  "celular": "987654321",
  "direccion": "Av. Principal 123, Lima",
  "correo": "juan.perez@example.com",
  "password": "SecurePassword123",
  "rol": "paciente"
}
```

#### Required Fields

- `dni` - National ID number (must be unique)
- `correo` - Email address (must be unique)
- `password` - Password (will be automatically hashed)

<Warning>
The system automatically converts `nombre`, `apellidoPaterno`, and `apellidoMaterno` to uppercase for consistency.
</Warning>

#### Password Security

Passwords are automatically hashed using bcrypt in the service layer (`src/service/usuario.service.js`):

```javascript
async crearUsuario(data) {
  data.password = hashedPassword(data.password)
  
  data.nombre = data.nombre.toUpperCase()
  data.apellidoPaterno = data.apellidoPaterno.toUpperCase()
  data.apellidoMaterno = data.apellidoMaterno.toUpperCase()
  
  return await usuarioDao.createUsuario(data)
}
```

## Retrieving Users

### Get All Users

```http
GET /api/user
```

Returns all users regardless of role.

### Get Users by Role

```http
GET /api/user?rol=paciente
```

Filter users by role. Valid values:
- `paciente`
- `fisioterapeuta`
- `admin`

**Example Response:**

```json
[
  {
    "_id": "507f1f77bcf86cd799439011",
    "nombre": "MARIA",
    "apellidoPaterno": "LOPEZ",
    "apellidoMaterno": "RODRIGUEZ",
    "dni": "87654321",
    "correo": "maria.lopez@example.com",
    "celular": "912345678",
    "estado": "activo",
    "rol": "paciente",
    "historialMedico": ["Cirugía de rodilla en 2020"],
    "alergias": ["Penicilina"]
  }
]
```

### Get All Patients

```http
GET /api/paciente
```

Returns all users with role "paciente".

### Get User by DNI

```http
GET /api/user/:dni
```

Retrieve a specific user by their national ID number.

**Example:**
```bash
curl http://localhost:8080/api/user/12345678
```

### Search Users by Name

```http
GET /api/user?name=Juan
```

Search for users whose name matches the query. The search converts the query to uppercase for matching.

## Updating Users

### Update User Information

#### Endpoint

```http
PUT /api/user
```

#### Request Body

```json
{
  "id": "507f1f77bcf86cd799439011",
  "nombre": "Juan Carlos",
  "celular": "999888777",
  "direccion": "Nueva Direccion 789"
}
```

The controller automatically converts name fields to uppercase:

```javascript
async updUser(req, res) {
  const body = req.body
  
  if (body.nombre) body.nombre = body.nombre.toUpperCase();
  if (body.apellidoPaterno) body.apellidoPaterno = body.apellidoPaterno.toUpperCase();
  if (body.apellidoMaterno) body.apellidoMaterno = body.apellidoMaterno.toUpperCase();
  
  const userUpdate = await usuarioService.modificarUsuario(body.id, body)
  
  return res.status(200).json({
    message: "Usuario cambiado",
    userUpdate
  })
}
```

### Update User Status

#### Endpoint

```http
PUT /api/user/status/:id
```

#### Request Body

```json
{
  "estado": "inactivo"
}
```

Change user status between `activo` and `inactivo`. This is useful for temporarily disabling user accounts without deleting them.

**Example:**
```bash
curl -X PUT http://localhost:8080/api/user/status/507f1f77bcf86cd799439011 \
  -H "Content-Type: application/json" \
  -d '{"estado": "inactivo"}'
```

## Password Management

### Change Password

#### Endpoint

```http
PUT /api/user/password
```

#### Request Body

```json
{
  "correo": "juan.perez@example.com",
  "password": "NewSecurePassword123"
}
```

The password is automatically hashed before storage:

```javascript
async updPass(req, res) {
  const body = req.body
  
  const userFind = await userModel.findOne({correo: body.correo})
  
  if(userFind) {
    const userUpdate = usuarioService.modificarContra(body.correo, body.password)
    
    logger.info("[UserController] Password updated successfully")
    
    return res.status(200).json({
      message: "Contraseña cambiada",
      userUpdate
    })
  }
  
  return res.status(404).json({
    error: "usuario no encontrado para el pass"
  })
}
```

<Warning>
Implement proper password reset verification (email tokens, OTP, etc.) in production. The current endpoint updates passwords based on email alone.
</Warning>

## Deleting Users

### Endpoint

```http
DELETE /api/user/:id
```

### Path Parameters

- `id` - User ID (MongoDB ObjectId)

**Example:**
```bash
curl -X DELETE http://localhost:8080/api/user/507f1f77bcf86cd799439011
```

<Warning>
Deleting a user permanently removes them from the database. Consider using status updates (`estado: "inactivo"`) instead for soft deletion.
</Warning>

## Complete User Creation Workflow

<Steps>
  <Step title="Prepare User Data">
    Collect all required user information:
    - Basic personal details (name, DNI, email)
    - Contact information (phone, address)
    - Password (will be hashed automatically)
    - Role-specific data (medical history for patients, credentials for therapists)
  </Step>
  
  <Step title="Submit Request">
    Send POST request to appropriate endpoint:
    - `/api/user` for generic users
    - `/api/paciente` for patients
    - Use appropriate endpoint for physiotherapists
    
    ```bash
    curl -X POST http://localhost:8080/api/paciente \
      -H "Content-Type: application/json" \
      -d @patient-data.json
    ```
  </Step>
  
  <Step title="Verify Creation">
    The API returns the created user object with:
    - Generated `_id`
    - Uppercased name fields
    - Hashed password
    - Default `estado: "activo"`
  </Step>
  
  <Step title="User Can Login">
    User can now authenticate using:
    - Email (correo) and password
    - Social authentication (Google/Facebook)
  </Step>
</Steps>

## Error Handling

<AccordionGroup>
  <Accordion title="Duplicate DNI or Email">
    Error occurs when trying to create a user with an existing DNI or email:
    
    ```json
    {
      "error": "E11000 duplicate key error"
    }
    ```
    
    Solution: Check if user exists before creation, or catch the error and return a user-friendly message.
  </Accordion>
  
  <Accordion title="Missing Required Fields">
    Mongoose validation error when required fields are missing:
    
    ```json
    {
      "error": "Path `dni` is required"
    }
    ```
    
    Solution: Ensure all required fields (dni, correo, password) are included in the request.
  </Accordion>
  
  <Accordion title="Invalid User ID">
    404 error when trying to update/delete non-existent user:
    
    ```json
    {
      "error": "usuario no encontrado"
    }
    ```
    
    Solution: Verify the user ID exists before attempting operations.
  </Accordion>
</AccordionGroup>
